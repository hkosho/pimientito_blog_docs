## 3. 回答ファイルの作成ルール

### 3.1 配置先と上書き運用

回答ファイルは以下に配置する:

```
security-report/responses/
```

**重要:** `security-report/responses/` 配下には、過去に Critical / High のみで作成した回答ファイル群が存在していたが、それらは既に退避済みである。本作業では **既存ファイルがある場合は上書き** し、全重要度を含めた新版として作成すること。

ディレクトリが存在しない場合は新規作成すること。

### 3.2 ファイル命名規則

章ごとに以下の命名で1ファイル作成:

```
response_ch{章番号2桁}_{章の英語簡略名}.md
```

例:
- `response_ch01_confirmed.md` (1章: 確認済み重大論点)
- `response_ch02_suspected.md` (2章: Repoから強く疑われる論点)
- `response_ch03_env_dependent.md` (3章: 本番環境依存)
- `response_ch04_config.md` (4章: 設定ファイル)
- `response_ch05_prisma.md` (5章: Prisma)
- `response_ch06_lib.md` (6章: lib層)
- `response_ch07_server_actions.md` (7章: Server Actions)
- `response_ch08_components.md` (8章: コンポーネント)
- `response_ch09_scripts.md` (9章: スクリプト)
- `response_ch10_backup.md` (10章: バックアップ)
- `response_ch11_grep.md` (11章: grep)
- `response_ch12_stack.md` (12章: 技術スタック固有)
- `response_ch13_spec.md` (13章: 仕様決定)

### 3.3 ファイルの記述形式

各回答ファイルは以下の構造で記述する。マトリクス形式を維持し、後で 3AI が突合判定を加筆できるようにする。

```markdown
# 回答ファイル: 第{N}章 {章タイトル}

**対象チェックリスト:** `security-report/checklist_security_verification.md` ver 1.0
**回答対象:** 第{N}章 {章タイトル}
**回答日:** {YYYY-MM-DD}
**回答者:** Claude Code
**参照コミット:** {`git rev-parse HEAD` の出力}
**処理対象:** 全重要度(Critical / High / Medium / Low)


---

## 回答マトリクス

| No. | 重要度 | 区分 | パス | ファイル名 | 質問(要約) | CC回答 | 評価(AI) | 判定根拠・備考 |
|---|---|---|---|---|---|---|---|---|
| {No.} | {重要度} | {区分} | {パス} | {ファイル名} | {質問の要約} | {回答本文} | {OK/NG/部分NG/判定不能} | {根拠となるコード引用、ファイル行番号、grep結果等} |

---

## 章サマリ

- **回答済項目数:** N件
  - Critical: N件
  - High: N件
  - Medium: N件
  - Low: N件
- **OK 件数:** N件
- **NG 件数:** N件
- **部分NG 件数:** N件
- **判定不能 件数:** N件
- **判定不能の主な理由:** (列挙)

## 参照したファイル一覧

- `path/to/file1.ts` (行番号 NN-MM)
- `path/to/file2.tsx`
- ...

## 実行した grep / コマンド一覧

- `git grep -n 'pattern1'`
- `find src -name '*.ts'`
- ...
```

### 3.4 列の記述ルール

- **質問(要約)**: チェックリスト原文の質問を 1 行に要約してよい(最大80文字程度)。要約により意味が変わる場合は原文をそのまま記載
- **CC回答**: 簡潔に事実のみ記載。コード引用が必要な場合は 3.5 のルールに従う
- **評価(AI)**: 必ず以下の4値のいずれかを記載
  - `OK`: 問題なし(該当箇所が安全である根拠を確認できた)
  - `NG`: 問題あり(脆弱性または不適切な実装を確認した)
  - `部分NG`: 一部に問題あり、または条件付きで問題化
  - `判定不能`: コード・設定だけでは判断できない(本番環境依存・仕様確認待ち・該当ファイル不存在等)
- **判定根拠・備考**: 必ず記載。OK判定でも「どのファイルのどの行で安全性を確認したか」を書くこと

### 3.5 コード引用のルール

- 引用は **必要最小限の行数**(原則10行以内)
- 引用元のファイルパスと行番号を必ず併記
- 機密情報(`.env.local` 内の実値、トークン値など)は引用しない。引用する場合はマスク(例: `SESSION_SECRET=***MASKED***`)
- 引用にはコードフェンスを使用し、言語識別子(```typescript 等)を付与

例:

```typescript
// frontend/src/lib/session.ts (行 12-18)
export const sessionOptions = {
  password: process.env.SESSION_SECRET!,
  cookieName: 'staff_session',
  cookieOptions: {
    httpOnly: true,
    secure: process.env.NODE_ENV === 'production',
    sameSite: 'lax',
    maxAge: 60 * 60 * 8,
  },
};
```

## 4. 判定の指針

### 4.1 判定不能とすべきケース

以下の場合は、推測で回答せず、必ず `判定不能` とすること。

- **本番環境依存**: リバースプロキシ設定、TLS 設定、ホスト OS、CDN/WAF、CI/CD 等、リポジトリ外の情報が必要なもの
- **仕様確定待ち**: 13章の項目、および「仕様決定者の判断が必要」と読み取れるもの
- **対象ファイル不存在**: チェックリストで指定されたファイルがリポジトリに存在しない場合
- **コード読解では確定できない動作**: 動的な振る舞い(例: 実際の HTTP 応答、実行時の挙動)に依存するもの
- **3章(2.C / Environment-dependent)の項目すべて**: 原則として全て判定不能扱い

「判定不能」とした場合、判定根拠欄に **判定不能の理由** を必ず記載すること。

### 4.2 推測の禁止

- 「おそらく〜だろう」「一般的には〜」「Next.js のデフォルトでは〜」といった **推測のみに基づく判定は禁止** する
- 一般論を補足として書くのは可。ただし「コード上の根拠なし」と明記すること
- ハルシネーションを避けるため、ファイルを実際に読まずに判定しないこと

### 4.3 判定が分かれるケース

- 「設定が複数ファイルにまたがり、組み合わせで判断する必要がある」場合は、関連ファイルすべてを引用したうえで判定する
- 「攻撃成立条件が成り立つか不明」な場合は `部分NG` とし、条件を備考欄に明記する

### 4.4 重要度別の判定姿勢

全重要度を対象とするが、判定の **厳密性** に差はつけない。Low の項目だからといって調査を簡略化しないこと。一方、回答の **詳細度** は重要度に応じて調整可。

- **Critical / High**: 判定根拠は詳細に。コード引用も必要に応じて積極的に
- **Medium**: 簡潔な根拠でよいが、ファイル特定は必須
- **Low**: 一行程度の根拠でも可

## 5. 進行ルール

### 5.1 進行単位

**章ごとに区切って進める**。1章を完了するごとに、以下を報告すること。

- 完成した回答ファイルのパス
- 当該章の項目数(全重要度別の内訳)
- 評価別の件数(OK / NG / 部分NG / 判定不能)
- 判定不能の主な理由(箇条書きで3〜5件)
- 章をまたぐ関連項目があった場合の連絡(例: 「2章 B-1-01 と 7章 G-1-01 は同一観点」)

### 5.2 推奨される進行順序

以下の順序を推奨する。重要度の高いものから、リポジトリの構造把握に役立つものを優先。

1. 9章 スクリプト・既存レポート確認 — 既存セキュリティレポートの内容把握
2. 4章 設定・環境ファイル確認 — リポジトリ全体の前提把握
3. 1章 確認済み重大論点 — 確認済みの問題の現状確認
4. 6章 認証・セキュリティライブラリ層 — 中核実装の確認
5. 7章 Server Actions — 重要操作の確認
6. 2章 Repoから強く疑われる論点 — 中核論点の確認
7. 5章 Prisma — データ層の確認
8. 8章 ページ・フォームコンポーネント
9. 11章 grep
10. 12章 技術スタック固有
11. 10章 バックアップ
12. 13章 仕様決定(全項目「判定不能」で記録のみ)
13. 3章 本番環境依存(全項目「判定不能」で記録のみ)

### 5.3 章をまたいだ重複処理の回避

複数の章で同じファイル(例: `frontend/src/lib/session.ts`)を確認する場合、コード引用は最初の章で行い、以降の章では「2章 B-4-01 参照」のように参照リンクで簡略化してよい。ただし評価(OK/NG/判定不能)は各項目で必ず記載すること。

## 6. 制約事項

### 6.1 やってはいけないこと

- **チェックリスト本体 (`security-report/checklist_security_verification.md`) の編集** ー 原本を保護するため、本ファイルへの書き込みは禁止
- **コードの修正・コミット** ー 本作業は静的検証であり、修正は別工程。たとえ明らかな脆弱性を発見しても、本作業中はコード修正・コミットを行わないこと
- **シークレット値の引用・出力** ー `.env.local` 等の実値を回答ファイルに書き込まないこと(マスク必須)
- **未読のファイルに基づく判定** ー 必ずファイルを読んでから判定すること
- **重要度による判定姿勢の変更** ー Low の項目だからといって調査を省略しないこと(4.4 参照)

### 6.2 やるべきこと

- 各章の作業前に、関連するファイルを `Read` 等で確認してから判定する
- 必要に応じて `git grep` 等を使用して横断検索を行う
- 回答ファイル冒頭に **参照したコミットハッシュ** を記録する(`git rev-parse HEAD` の出力)
- 章ごとの完了時に、人間に簡潔な進捗報告を行う
- `security-report/responses/` 配下の **既存ファイルは上書き** すること(退避済みのため復元不要)

## 7. 想定される困難と対処

### 7.1 コンテキスト長の不足

ファイルが多数にわたる場合、1セッションで全章を処理するとコンテキスト長を超える可能性がある。その場合は以下に従う:

- セッションを章ごとに区切る
- 各セッション開始時に「前章までの完了状況」を冒頭で確認する
- 章をまたぐ重複処理は 5.3 のルールで簡略化する
- それでも不足する場合は、章を「Critical/High 部分」と「Medium/Low 部分」の2セッションに分割可。ただし出力ファイルは1つにまとめること

### 7.2 関連ファイルの大量性

例えば 11章(grep)は対象が広範。grep 結果が大量にヒットした場合は:

- 件数のみ集計し、上位N件(目安: 10件)のみ詳細引用
- 残りは「他NN件は別途精査」と備考に記載
- 件数だけでも判定の参考になるため、必ず数値は記録

### 7.3 既存レポートとの重複

`security-report/security-report.md` 等の既存レポートで既に指摘済みの項目がある場合:

- 重複を回避し、回答欄に「既存レポート `security-report.md` セクションXX で指摘済み」と記載
- 既存レポート以降に変更があった場合は、現状との差分を回答する

### 7.4 退避済みの旧回答ファイルとの差異検証

過去に Critical / High のみで作成した回答ファイル群は退避済みであり、本作業の対象外である。新版作成時に旧版を参照する必要はない。ただし、もし退避前のファイルがディレクトリに残存している場合は、上書き保存することで問題ない(差分比較は人間が後工程で実施する)。

## 8. 完了の定義

本作業は、以下が揃った時点で「完了」とする。

1. `security-report/responses/` 配下に、全章の回答ファイル(13本)が作成されている(既存ファイルは上書き)
2. 各回答ファイルに、対象章の全項目(全重要度)の行が埋まっている(空欄なし)
3. 各回答ファイルの「章サマリ」欄が記入されている(重要度別の内訳含む)
4. 章ごとに人間への進捗報告が行われている
5. 全章完了後、最終サマリ(全章の OK/NG/判定不能の集計、重要度別内訳含む)が報告されている

完了後、人間側で以下の作業に進む(本指示書の対象外):

- OSSツール(OWASP ZAP / sqlmap / Nmap / Semgrep / Trivy / OpenVAS)による動的検証
- 突合判定および修正方針決定
- 修正・再検証

## 10. 補足: 本作業の位置付け

本作業は、ブログ『生成AIと作るローカルLLMサーバ奮闘記』のWebサーバ構築・脆弱性診断編における、AIと人間の協働実験の一部である。

- あなた(Claude Code)の回答 = **静的検証の結果**
- 後工程の人間 + OSSツール = **動的検証の結果**
- ChatGPT / Claude / Gemini = **両結果の突合判定**

本実験では、**AI が見落とした脆弱性(OK-NG)** および **AI が過剰警告した項目(NG-OK)** が最重要のエビデンスとなる。したがって、あなたが「判定不能」と正直に記録することは、無理に判定するよりも価値が高い。**正確性を優先し、ハルシネーションを避けること**。
